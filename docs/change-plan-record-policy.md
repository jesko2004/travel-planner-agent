# 修改方案：持久化开发记录规则

> 状态：已确认（用户于 2026-07-18 明确要求）

## 1. 目标与非目标

- 目标：让任何后续 Agent 在上下文重置后仍能知道，项目文件发生变更时必须追加 `docs/record.txt`。
- 用户场景：用户希望回看每次代码生成或文件变更的实现摘要、疑难点和解决办法。
- 成功标准：规则进入 `limitation.md`；记录格式统一为 `YYYY-MM-DD：内容`；本次变更本身有记录。
- 本次不做：不保存完整提示词、逐字内部思维、密钥、私人数据或完整模型响应；不增加运行时日志或外部服务。

## 2. 规范影响

- 是否需要修改 `limitation.md`：是，在“变更治理与决策记录”中增加强制追加规则，并索引记录文件。
- 是否新增 MCP、数据源或外部写操作：否。
- 是否涉及敏感数据：否；规则明确禁止敏感内容进入记录。
- 数据可信度与来源：记录内容只描述本地项目变更和实际执行的验证。

## 3. 设计

- 页面变化：无。
- 工作流变化：开发流程在最终答复前追加一条开发记录。
- 数据模型或 SQLite 变化：无。
- 失败与降级：无法写入记录时必须向用户说明，不得静默声称记录完成。
- 兼容与迁移：新增 `docs/record.txt`，并将当前会话中已有记录整理为“日期+内容”；不涉及运行时数据或 schema 迁移。
- 回滚：通过新的 PR 撤销 `limitation.md` 中的强制规则与索引，并删除本次新增的政策文档和记录文件；如需保留历史记录，回滚前先转存到用户确认的位置。

## 4. 实现步骤

1. 在 `limitation.md` 增加开发记录强制规则和 `docs/record.txt` 索引。
2. 新增 `docs/record.txt`，并采用 `YYYY-MM-DD：内容` 格式。
3. 执行文档一致性和 Git 差异检查，并追加本次记录。

## 5. 测试与验收

- 单元测试：运行现有离线测试，确认文档变更未影响项目；不新增运行时代码测试。
- 集成测试：不涉及外部服务，未运行真实 DeepSeek/高德集成测试。
- 手工场景：检查规则可从 `AGENTS.md -> limitation.md -> docs/record.txt` 被发现；检查每条记录格式。
- 隐私与密钥检查：确认新增文本不含凭证或私人数据。
