# 工作流、降级与验收

## 本轮基础闭环

当前 PR 实现可以离线测试的可信基础，执行顺序为：

输入结构校验与敏感扫描 → 构造高德最小化查询 → 构造 DeepSeek 最小化规划数据 → 草案映射 → 路线补全 → 构造 `ValidationContext` → 确定性校验与状态派生 → SQLite 事务保存和回读 → 页面/ICS。

外部调用前必须完成扫描；发现敏感数据后不允许调用假模型、假 MCP 或 SQLite。扫描同时使用类别规则和当前已加载凭证的精确值 denylist，不回显命中值。共享行程对象按顺序更新，不依据自然语言中的“成功”“已验证”等字样判断步骤结果。

完整 10 阶段状态机、调用预算/取消机制、真实高德工具调用记录适配器与完整 UI/ICS 加固不在本轮范围，后续单独实现。

## 确定性校验

校验器使用显式 `ValidationContext`，不再用单个 `amap_available` 布尔值代表可信度。固定检查包括：

- 行程不能为空，活动日期必须与父 `DayPlan` 和请求范围一致。
- 活动开始与结束均位于每日时间窗内，且同日活动不重叠。
- POI ID 不重复、坐标合法、坐标系为 GCJ-02，并位于批准候选白名单。
- 必需证据的来源、类型、查询时间和过期时间完整且在有效期内。
- 每条路线必须精确连接对应相邻活动，并具有正距离、正时长和有效证据。
- 下一活动开始时间必须容纳路线时长和至少 15 分钟缓冲。
- 锁定活动不得被删除、改期、改时、换地点或解除锁定。

未知 POI、锁定活动变更、路线不可达以及结构/来源错误会生成 error 并阻止 `verified`；步行或预算超限生成 warning，但仍会醒目展示。

## 状态与降级

- `draft`：任一必需阶段尚未完成。
- `unverified`：存在结构、业务或来源错误，或缺少可信工具调用记录。
- `expired`：必需阶段完成且无其他错误，唯一问题是所需证据过期。
- `verified`：必需阶段完成、零 error，且所有必需证据当前有效并可追溯到批准的真实工具记录。

当前没有真实高德工具调用记录适配器，因而实际高德/模型结果默认保持 `unverified`。这是一项有意的可信降级，不影响用户查看草案。

DeepSeek 不可用时停止生成；高德不可用时只生成未验证草案；单条路线失败时保留活动但产生 error；SQLite 初始化、迁移、保存或回读失败时保留当前会话结果并明确显示“未保存”，不写入备用位置。

## 持久化与恢复

SQLite 使用 `PRAGMA user_version=1`。检测到旧版 `0` 时，先解析并执行敏感扫描，再通过 SQLite backup API 在相邻 `backups/` 创建带时间标识的 `.db` 备份，随后在事务中迁移并逐 ID 回读。失败时事务回滚、原库保留；恢复方式是使用自动备份，或回滚本次迁移代码后读取原库。

历史列表与详情不是静态可信快照。回读会按当前北京时间重新执行证据与业务校验，并限制为只降不升：旧 `verified` 可因 TTL 到期成为 `expired`，旧 `unverified` 不会在没有新工具证据时被提升。

## 验收基线

- 固定 `Asia/Shanghai` 时钟覆盖全部 TTL 和精确过期边界。
- 输入长度、列表、预算、儿童年龄、额外字段及敏感信息类别均有反例测试。
- 空行程、日期错位、结束越界、坐标、POI 白名单/重复、路线端点/时长/缓冲和锁定活动均有回归测试。
- 旧 SQLite fixture 覆盖备份、`0→1` 迁移、回读、更新时间及失败回滚。
- 无真实工具调用记录的数据永远不能成为 `verified`，但页面仍可展示醒目的未验证草案。
- `compileall`、完整 `pytest` 和 Git 凭证模式扫描在 Python 3.10/3.12 CI 执行。
- 酒店不出现未经正式来源验证的价格、库存或取消承诺；ICS 从结构化活动时间生成。
